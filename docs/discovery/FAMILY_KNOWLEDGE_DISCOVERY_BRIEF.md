# Family Knowledge — Discovery Brief

> **Статус:** Discovery (не build)
> **Дата:** 2026-05-20
> **Трек:** Параллельный sidecar к Wave 3 / Integration Baseline

---

## Зачем это нужно

Сейчас у пользователя нет единого места, где семья накапливает и передаёт знания:
- правила и договорённости разбросаны по разным разделам
- воспоминания хранятся в альбоме без контекста
- предпочтения и интересы членов семьи нигде не собраны
- советы и рекомендации Домового не опираются на семейный контекст

Family Knowledge — это knowledge layer, который:
1. даёт пользователю инструмент фиксации «того, что важно знать о нашей семье»
2. обогащает Домового и другие ИИ-модули реальным семейным контекстом
3. становится основой будущей Family Library

## Jobs To Be Done

| # | Job | Кто | Когда |
|---|-----|-----|-------|
| J1 | Зафиксировать правило/договорённость семьи | Любой взрослый | Когда договорились о чём-то важном |
| J2 | Найти информацию о члене семьи (предпочтения, привычки, аллергии) | Любой взрослый | Перед важным решением или событием |
| J3 | Передать знание следующему поколению | Старшее поколение | Осмысление |
| J4 | Дать Домовому контекст для персонализированного ответа | Система | При каждом AI-запросе |
| J5 | Найти воспоминание, связанное с конкретным человеком или датой | Любой | Ностальгия / планирование |

## Типы объектов знания (Knowledge Objects)

| Тип | Описание | Примеры |
|-----|----------|---------|
| `fact` | Факт о семье/члене | Аллергия на орехи, размер обуви, любимое блюдо |
| `rule` | Правило или договорённость | Телефоны до 21:00, субботний ужин вместе |
| `memory` | Воспоминание с контекстом | Поездка в 2019, как познакомились |
| `profile_preference` | Предпочтение/интерес члена | Любит детективы, занимается плаванием |
| `recommendation` | Совет или рекомендация | Хороший врач, ресторан для особых случаев |
| `document_ref` | Ссылка на документ/артефакт | Полис, рецепт, инструкция |

---

## Ownership (чья это запись)

| Уровень | Описание |
|---------|----------|
| `personal` | Принадлежит конкретному члену семьи |
| `family_shared` | Принадлежит семье целиком |
| `system` | Создано/предложено Домовым или инференцией |

---

## Trust / Source (источник знания)

| Источник | Описание |
|----------|----------|
| `user_entered` | Введено пользователем вручную |
| `inferred` | Выведено системой на основе активности |
| `ai_suggested` | Предложено Домовым, не подтверждено |
| `ai_suggested_verified` | Предложено Домовым и подтверждено пользователем |

**Принцип:** всё что ниже `user_entered` — не может быть показано как «факт» без явного подтверждения.

---

## Visibility (видимость)

| Режим | Описание |
|-------|----------|
| `private` | Только создавший + те, кому явно открыт доступ |
| `family_shared` | Все члены семьи |
| `public` | Будущее (для Family Library, вне скоупа MVP) |

## MVP Boundaries

### Входит в MVP

- Базовые типы: `fact`, `rule`, `memory`, `profile_preference`
- Ownership: `personal` и `family_shared`
- Source: `user_entered` только (никакого AI inference в MVP)
- Visibility: `private` и `family_shared`
- CRUD: создать, просмотреть, редактировать, удалить
- Привязка к члену семьи (through family_member_id)
- Привязка к категории/тегу

### Не входит в MVP

- `recommendation`, `document_ref` — отдельный трек
- AI inference и `ai_suggested` — нужна RAG инфраструктура
- `public` visibility — ожидает Family Library
- Полнотекстовый поиск по базе знаний — нужен отдельный индекс
- Semantic similarity / embeddings — нужен RAG

---

## Связь с будущими треками

```
Family Knowledge MVP
  ↓ knowledge objects (facts, rules, preferences)
  
Family Library V1
  ↓ rich content, articles, recommendations
  
Домовой grounded RAG
  ↓ векторизация knowledge objects
  ↓ retrieval при каждом AI-запросе
  ↓ personalized responses на основе реального семейного контекста
```

## Открытые вопросы (для следующего этапа)

1. **Где живёт UI?** — отдельный раздел `/family-knowledge`, или distributed по хабам?
2. **Как связать с профилями?** — через `family_member_id` или отдельная таблица preferences?
3. **Гранулярность privacy** — достаточно ли binary private/shared или нужны ACL?
4. **Интеграция с Memories** — дублирование или complementary?
5. **Формат для RAG** — plain text embedding или structured JSON с метаданными?
